# 数据库设计

> 本文档定义禾大 Croda 情报订阅 Agent 的 SQLite 数据模型。设计目标是支持文章采集、正文抽取、结构化打标、证据追踪、客户 watchlist 匹配、活字典维护和月报生成。

---

## 一、设计目标

1. **服务禾大情报场景**：围绕竞品、客户品牌、代理商、成分、技术、功效、法规和市场活动组织数据。
2. **字段 vs 标签分离**：字段由入库脚本写在文章/批次表；标签由 Agent 写在 `article_tags`（见第六节）。
3. **支持多来源采集**：微信公众号、竞品官网、行业媒体、展会和法规源进入同一文章库（暂不接入专业数据库订阅）。
4. **保留证据链**：每个正式标签必须能追溯到一条客观证据 `evidence_text`（取自标题/摘要/正文）。
5. **公司自由抽取 + watchlist 匹配**：公司名由 Agent 自由抽取，客户关注名单作为配置与匹配层，不作为 taxonomy；公司角色由 watchlist 匹配得出。
6. **字典表 + 实时活字典**：`taxonomy_terms` 作为 picklist 单一事实来源；开放字段（`ingredient_technology`/`product_application`/`functional_claim`）允许 `other:<slug>` 内联打标，按频次驱动晋升。
7. **支持月报生成**：数据库不仅服务打标，也要能聚合出 TOP10、竞品矩阵、成分趋势、技术创新、活动汇总和附录链接。

## 二、推荐数据流

```text
taxonomy_terms (字典表/picklist，种子+晋升)
source_profiles + watchlist_entities
→ article_import_runs（含抓取/失败日志）
→ articles（字段：来源、URL、时间、语言、地区、来源性质、抓取方式）
→ article_contents（正文 + extraction_status + raw_tags）
→ article_tag_status
→ v_articles_for_tagging
→ tagging_runs / tagging_attempts
→ article_tags（仅 Tag↔文章关联）+ tag_evidence（仅 evidence_text）
→ inline_other_terms + suggested_new_tags + tag_review_queue
→（活字典晋升回写 taxonomy_terms）
→ v_monthly_report_articles / report_runs
→ HTML 月报 + JSON 摘要
```

> **字段 vs 标签落位（关键）：** 来源、URL、时间、语言、地区、来源性质、抓取方式等**字段（属性）**由入库脚本写在 `articles` / `article_contents` / `article_import_runs` 上；**标签（Agent 判断）**写在 `article_tags`。`article_tags` 只保留 Tag 与文章的关联，不混入抓取脚本、来源配置等"硬编码"属性。详见第六节"数据流与字段归位"。

| `rss_url` | TEXT | RSS 地址，可为空 |
| `ingest_method` | TEXT NOT NULL | `wechat_account`、`direct_rss`、`google_news_rss`、`web_scraper`、`manual_or_reference` |
| `market_region_json` | TEXT | 来源默认覆盖地区，仅作提示 |
| `language_json` | TEXT | 来源语言 |

| `created_at` / `updated_at` | TEXT | ISO datetime |

### 3.2 客户关注实体表：`watchlist_entities`

客户 Excel 中的行业媒体、竞品、代理商、MNC 品牌、国内品牌进入配置表。它用于匹配和筛选，不限制 Agent 抽取公司。

| 字段 | 类型 | 说明 |
|------|------|------|
| `entity_id` | TEXT PRIMARY KEY | 稳定实体 ID |
| `display_name` | TEXT NOT NULL | 显示名 |
| `display_name_normalized` | TEXT | 归一化键，用于与 `company` 匹配 |
| `aliases_json` | TEXT | 别名、中英文名、品牌名，用于**规范化**（如 `欧莱雅`=`L'Oréal`、`禾大`=`Croda`） |
| `entity_role` | TEXT NOT NULL | 配置项（非打标标签）：`self`、`competitor`、`customer`、`channel_partner`、`ecosystem` |
| `market_region_json` | TEXT | `china`、`global`、`europe` 等（用于还原"国际/国内"） |
| `value_chain_stage` | TEXT | 可选，公司主营链位置 |
| `source_file` | TEXT | 来源 Excel 或人工配置 |
| `status` | TEXT | `active`、`paused`、`archived` |
| `created_at` / `updated_at` | TEXT | ISO datetime |

> Watchlist 是**客户配置**：公司名不进 `suggested_new_tags`、不回写字典词表；由客户独立维护、定期同步增删。后端用 `display_name_normalized` + `aliases_json` 把 Agent 抽取的 `company` 归一化匹配，再读出 `entity_role`/`market_region`。

### 3.7 打标批次与尝试：`tagging_runs` / `tagging_attempts`

`tagging_runs` 记录一批任务，`tagging_attempts` 记录每篇文章每次尝试。

关键字段包括 `tag_run_id`、`agent_name`、`model_name`、`dictionary_version`、`prompt_version`、`run_type`、`status`、`metrics_json`、`attempt_number`、`validation_result`、`post_processing_flags_json`。

### 3.8 标签-文章关联表（"字典文章关联表"）：`article_tags`

> **这就是用户问的"字典文章关联表"。** 它是打完标签后生成的 Tag↔文章关联表，**只保留 Tag 与文章的关联信息**：哪篇文章被打了哪个字段的哪个值。抓取脚本、来源配置、字典版本、批次等"非关联"信息**不放这里**——批次/来源属性在 `articles` / `article_import_runs`，溯源放在 `tag_evidence` / `tagging_attempts`。

| 字段 | 类型 | 说明 |
|------|------|------|
| `article_tag_id` | TEXT PRIMARY KEY | 关联记录 ID |
| `article_id` | TEXT NOT NULL | 关联文章 |
| `field` | TEXT NOT NULL | 标签字段（如 `ingredient_technology`） |
| `label` | TEXT NOT NULL | 标签值，开放字段允许 `other:<slug>` |
| `display_name` | TEXT | 可选显示名，用于 `other:<slug>` 或 company |
| `status` | TEXT NOT NULL | `active`、`superseded`（仅维护关联生命周期） |

推荐索引：`UNIQUE(article_id, field, label)`、`INDEX(field, label, status)`。

> 溯源（哪次批次、什么字典版本、AI/人工/回填）通过 `tag_evidence.tag_run_id` 与 `tagging_attempts` 关联取得，不在本关联表内冗余。`field`/`label` 应能与 `taxonomy_terms`（字典表）join 校验，开放字段的 `other:<slug>` 在晋升前不入 `taxonomy_terms`。

### 3.9 标签证据表：`tag_evidence`

| 字段 | 类型 | 说明 |
|------|------|------|
| `evidence_id` | TEXT PRIMARY KEY | 证据 ID |
| `article_tag_id` | TEXT | 关联正式标签 |
| `article_id` | TEXT NOT NULL | 关联文章 |
| `tag_run_id` | TEXT NOT NULL | 关联批次 |
| `field` | TEXT NOT NULL | 标签字段 |
| `label` | TEXT NOT NULL | 标签值 |
| `extracted_name` | TEXT | 原文显示名，主要用于 `other:<slug>` |
| `evidence_text` | TEXT NOT NULL | 一条客观证据原文（取自标题/摘要/正文） |
| `created_at` | TEXT NOT NULL | 写入时间 |

> 已精简：移除 `evidence_field`（证据来源字段）与 `reason`（判断理由），只留客观证据 `evidence_text`，与字典 §7.3 一致。

### 3.10 字典表（picklist）：`taxonomy_terms`

> **这就是用户问的"字典表"。当前数据库设计原本没有它**——之前 taxonomy 只活在 Markdown 字典 + `dictionary_version` 字符串 + `validate_tags.py` 里，没有可查询的 picklist 表。现新增 `taxonomy_terms` 作为**字段可选值（picklist）的单一事实来源**。

| 字段 | 类型 | 说明 |
|------|------|------|
| `term_id` | TEXT PRIMARY KEY | 词条 ID |
| `field` | TEXT NOT NULL | 所属标签字段，如 `primary_story_type`、`ingredient_technology` |
| `label` | TEXT NOT NULL | 受控值（snake_case）；晋升后的原 `other:` 也写入这里 |
| `display_name_zh` | TEXT | 中文名 |
| `display_name_en` | TEXT | 英文名 |
| `parent_label` | TEXT | 可选父类 |
| `status` | TEXT NOT NULL | `active`、`candidate`、`merged`、`rejected`、`deprecated` |
| `origin` | TEXT NOT NULL | `seed`（字典种子）、`promoted_inline_other`、`promoted_suggested`、`manual` |
| `is_open_field` | INTEGER | 该字段是否允许 `other:`（1=开放：ingredient_technology/product_application/functional_claim） |
| `first_seen_at` / `promoted_at` | TEXT | 时间 |
| `dictionary_version` | TEXT | 入库时字典版本 |
| `note` | TEXT | 备注 |

推荐索引：`UNIQUE(field, label)`、`INDEX(field, status)`。

**picklist 创建与维护机制：**
- **创建（种子）**：从 `标签字段字典.md` 各字段词表导入，`origin='seed'`、`status='active'`。某字段当前可选值即 `SELECT label FROM taxonomy_terms WHERE field=? AND status='active'`。
- **晋升（活字典）**：开放字段的高频 `other:<slug>`（见 `inline_other_terms` + `v_inline_other_frequency`）或封闭字段的 `suggested_new_tags` 经人工通过后，写入本表（`origin='promoted_*'`、`status='active'`），并可触发历史 `article_tags.other:<slug>` 归一化为新正式 `label`。
- **校验**：`validate_tags.py` / runner 以本表为准判定合法 label；开放字段额外放行 `other:<slug>`。

## 四、推荐视图

### `v_articles_for_tagging`

稳定提供给打标 runner 的文章包：文章基础字段、正文、来源默认信息、抽取状态、打标状态、raw tags。

### `v_watchlist_matches`

根据 `company` 标签和 `watchlist_entities.aliases_json` 生成匹配结果，用于区分竞品、客户、代理商和生态伙伴。

### `v_monthly_report_articles`

按月份、链接状态、标签、公司角色和复核状态聚合，服务月报生成器。

### `v_inline_other_frequency`

统计开放字段 `other:<slug>`（`ingredient_technology` / `product_application` / `functional_claim`）在窗口期内的文章数、来源数、最新出现时间和代表证据，服务人工批量晋升到 `taxonomy_terms`。

### `v_taxonomy_picklist`

按 `field` 输出当前 `status='active'` 的 picklist，供 runner / validator / 月报筛选器读取。

## 五、关键约束

- `company` 是自由实体，不受 watchlist 限制；公司**角色**由 `v_watchlist_matches` 匹配得出，不在打标阶段判定（已无 `entity_role` 标签）。
- 开放字段（`ingredient_technology`、`product_application`、`functional_claim`）允许 `other:<slug>`；封闭枚举字段（`primary_story_type`、`value_chain_stage`）不得发明正式标签。
- `article_tags` 只存 Tag↔文章关联；字段（属性）与溯源不混入。
- 字段（来源、语言、地区、来源性质、抓取方式）由入库脚本写在 `articles`/`article_contents`，打标 Agent 不重算。
- 无 `evidence_text` 的正式标签不得写入 `article_tags`。
- 抓取失败的文章不入库；失败信息只留在 `article_import_runs.error_log`（不设 `access_status`/`no_rss`/`blocked_*` 等字段）。
- 技术失败、内容不足、taxonomy 缺口和业务歧义必须区分处理。

## 六、数据流与字段归位

按用户要求，明确"哪些字段从 RSS 一路保留进库""哪些只留在抓取日志""字段 vs 标签如何落位"。

### 6.1 从 RSS → 数据库（保留进库）

抓取脚本在入库前判定并写入 JSON，落到 `articles` / `article_contents`：

| 数据 | 来源阶段 | 落库表 | 性质 |
|------|----------|--------|------|
| 标题、摘要、URL/canonical_url、发布时间、作者、原始 JSON | RSS/抓取 | `articles` | 字段 |
| 抓取时间 `fetched_at`、来源 `source_key`/来源名快照 | 抓取 | `articles` | 字段 |
| 抓取方式 `ingest_method` | 来源配置/URL | `articles` 或 `article_import_runs` | 字段 |
| 语言 `content_language` | 语言检测脚本 | `articles` | 字段 |
| 地区 `market_region` | 来源配置/地名扫描 | `articles`（显式证据时） | 字段 |
| 来源性质 `source_nature` | 来源配置/署名 | `articles` | 字段 |
| 正文 `content`、`content_length`、`content_hash`、`raw_tags` | 正文抽取 | `article_contents` | 字段 |
| 抽取状态 `extraction_status`（仅 success/partial） | 正文抽取 | `article_contents` | 字段 |
| 链接有效性 `link_status` | **月报生成阶段**校验 | `articles` | 字段（后置） |
| Agent 判断标签（primary_story_type 等 6 个） | 打标阶段 | `article_tags` | 标签 |

### 6.2 只留在抓取日志、不进库

| 数据 | 原因 | 去向 |
|------|------|------|
| 抓取失败/超时/WAF 阻断/401/无 RSS/无可用源 | 失败文章不产生 article 行 | `article_import_runs.error_log`（抓取日志） |
| 重试次数、HTTP 状态码、临时报错 | 运维信息，非情报 | 抓取日志 |
| 专业数据库（SpecialChem 等）查询 | 当前未接入订阅 | 不实现 |
| 来源可达性 `access_status` | 来源运维状态，非文章属性 | 来源配置/抓取日志 |

### 6.3 字段 vs 标签的落位规则
- **字段（属性）**：能由脚本/规则在入库前确定 → 写 `articles`/`article_contents`/`article_import_runs`，打标 Agent 不重算。
- **标签（语义判断）**：需读懂文章 → 由 Agent 输出 → 只写 `article_tags`（Tag↔文章关联）+ `tag_evidence`（证据）。
- **硬编码/配置**（抓取脚本、来源配置、字典版本、批次号）：放来源表/批次表/审计表，**不进** `article_tags`。
- **picklist（字典）**：`taxonomy_terms` 是字段可选值的单一事实来源；开放字段的字典外值先以 `other:<slug>` 落 `article_tags` 与 `inline_other_terms`，晋升后回写 `taxonomy_terms`。

### 6.4 两个被问到的表小结
- **字典表 = `taxonomy_terms`（3.10，新增）**：原设计没有，现补上，作为 picklist 单一事实来源 + 活字典晋升落点。
- **字典文章关联表 = `article_tags`（3.8，已精简）**：打标后生成，**只保留 Tag↔文章关联**（article_id + field + label + display_name + status），溯源与属性不混入。
