## 007 | 2026-05-29 | 数据库设计

**类型：** 数据库设计  
**决策方：** User + Codex  
**状态：** Confirmed

**内容：**

根据用户要求，基于新标签字典反推 SQLite 数据库结构，新增 `dev/docs/数据库设计.md`。

核心设计决策：

- 不直接复用 `reference/wewe-rss/` 的 Feed/Article schema；该参考结构适合 RSS 服务展示，但缺少 AI 打标需要的标签关系、证据、候选标签、Other 审核和公司 watchlist。
- 新建本项目自己的 SQLite schema，分为来源、RSS 导入批次、文章基础信息、正文、打标状态、打标批次、正式标签、标签证据、候选/Other 审核、公司 watchlist 和词表版本。
- `articles` 表只保存文章基础事实；多值标签不写入 `articles`，统一写入 `article_tags`。
- `tag_evidence` 独立保存每个标签的证据，支持人工复核和重打标追溯。
- `other_candidates` 独立服务活字典维护，候选标签经人工 qualification 后才能进入正式词表。
- `company_watchlist` 独立服务 `company_tags`，支持客户指定公司和历史回填。
- 新增 `v_articles_for_tagging` 作为 OpenClaw 读取待打标文章的稳定视图。

**原因：**

下一步需要将 RSS 原始 JSON 入库，并让 OpenClaw 访问 SQLite 完成打标签。若把标签直接塞进文章表或 JSON blob，后续筛选、重打标、证据审计和活字典维护都会困难。因此本次设计选择规范化存储正式标签与证据，同时保留原始 JSON 和正文，兼顾可追溯性与 Agent 批量处理效率。

**后续修订：**

2026-06-03：008 变更废弃基于 AI 自评 confidence 的低置信候选口径，改为 `suggested_new_tags`、`tag_review_queue`、schema validation、evidence validation 和 post-processing。

**类型：** 数据规范  
**决策方：** User + Codex  
**状态：** Confirmed

**内容：**

根据用户反馈，重构 `dev/docs/标签字段字典.md` 并同步 `newsletter-tagging/`：

- 将标签体系改为两层：基础数据字段标签与 Agent 判断标签。
- 基础数据字段包括来源、抓取方式、抽取状态、语言、地区、来源性质等，优先由代码或轻量规则处理。
- Agent 判断标签压缩为 MECE 维度：`primary_story_type`、`industry_segment`、`product_application`、`strategic_driver`、`value_chain_stage` 和自由实体 `company`。
- 取消公司预设 watchlist。Agent 在文章事件确实与公司有关时自由抽取公司显示名，写入 JSON / 数据库，不回写字典词表。
