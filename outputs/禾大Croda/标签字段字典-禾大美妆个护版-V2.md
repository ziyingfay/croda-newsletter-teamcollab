# 标签字段字典 · 禾大 Croda 美妆个护原料版

> 本文档定义禾大 Croda 情报订阅 / newsletter 的文章字段与标签体系。当前版本（V2）在两层 MECE 基础上，进一步**严格区分"字段（属性 fields）"与"标签（tags）"**，并精简 Agent 判断标签。
>
> - 配套需求分析：`outputs/禾大Croda/禾大Croda-Newsletter需求分析报告.md`
> - 配套数据库与数据流：`dev/docs/数据库设计.md`
> - 父项目食品饮料字典已归档：`reference/legacy-food-beverage/标签字段字典-食品饮料版.md`
> - schema 命名：`newsletter-tagging/croda-beauty-v1`（字典版本 `croda-beauty-2026-06-04-v2`）

---

## 设计原则

1. **字段 vs 标签分离**：
   - **字段（属性 fields）**：文章 ID、来源、语言、抓取方式、URL、时间、来源分类等，由**抓取/清洗/入库脚本在入库前统一写入标准文章 JSON**。其中一部分来自 RSS 原始字段，一部分来自 `source_profiles` 媒体源配置，一部分需要脚本判断或格式规范化。打标 Agent **不重算、不补全**这些字段。它们存在**文章表 / 导入批次表**，**不进入"标签-文章关联表"**。
   - **标签（tags）**：需要读懂文章语义才能判断的维度，由 OpenClaw Agent 判断，写入"标签-文章关联表"。
2. **MECE 优先**：每个标签字段一个独立维度，相同含义不跨字段重复。
3. **不盲目细分**：不能稳定提升筛选价值的维度删除。本版已删除 `industry_segment`、`strategic_driver`、`entity_role`。
4. **公司不入字典词表**：公司名由 Agent 自由抽取写入 `company`；客户/竞品**监测名单（Watchlist）**是客户配置（见第六节），用于归一化与匹配，不作为 taxonomy。
5. **证据驱动，输出精简**：每个标签给一条**客观证据原文 `evidence_text`**即可，不再要求 `evidence_field` 与 `reason`。
6. **活字典（Picklist）**：开放维度（`ingredient_technology`、`product_application`、`functional_claim`）字典外的值由 Agent 实时写 `other:<名>`，频次驱动人工晋升进正式词表（见第五节）。

## 一、两层字段结构

### 1.1 基础数据字段（属性 fields，入库脚本写入，Agent 不补全）

> 这些字段在 RSS/公众号/官网抓取后，由脚本依据**RSS 原始字段、`source_profiles` 媒体源配置、URL、元数据、语言检测、正文抽取结果**写进标准文章 JSON。抓取失败的文章无法入库，因此数据库中**不需要**"需订阅 / 无 RSS / 无可用源 / WAF 阻断 / 401 认证"等失败状态——这些只留在**抓取日志**里。
>
> MVP 数据流中的主键关系：`raw_ref` 用于回溯原始抓取记录；`article_id` 是 `rss_clean.json`、`tagging.json`、`report_content.json` 之间的稳定串联键；`source_key/source_name` 用于命中 `source_profiles`，自动带出 `source`、`source_list`、`ingest_method`、默认 `market_region` 和默认 `language`。

| 字段 | 类型 | RSS/抓取结果是否自带 | 脚本处理方式（入库前） | 存放位置 | 备注 |
|------|------|----------------------|------------------------|----------|------|
| `article_id` | string | 否 | 清洗脚本生成；建议基于 `canonical_url` 哈希，缺 `canonical_url` 时用规范化 `url`，再缺时用 `source_key + title + published_at` 兜底 | articles；`tagging.json`；`report_content.json` | **MVP 主串联键**，用于把原文、标签和报告内容一一对应 |
| `raw_ref` | object/null | 否 | 清洗脚本写入原始文件和条目位置，如 `raw_file`、`raw_item_index`、原始平台 ID | articles 或 run log | 轻量回溯原始 RSS/公众号记录，不给 Agent 当标签判断依据 |
| `title` | string | 是 | 从 RSS/抓取结果读取，做去空格、HTML entity 解码、异常换行清理 | articles | Agent 与报告都需要 |
| `summary` | string/null | 可能有 | 从 RSS summary/description 或正文前段生成，做格式清理 | articles | 可为空，但标题+正文至少应足够判断 |
| `content` | string/null | 通常否 | 正文抽取脚本写入，清理导航、广告和明显无关文本 | article_contents / `rss_clean.json` | 小龙虾 1 的主要阅读输入 |
| `content_length` | integer | 否 | 脚本按清洗后 `content` 计算 | article_contents / `rss_clean.json` | 用于判断正文是否过短或需要复核 |
| `url` | string | 是 | 从 RSS `link` / `url` / 抓取入口读取，做去空格、解跳转、去跟踪参数等格式规范化 | articles | 属于 RSS 自带字段的标准化 |
| `canonical_url` | string/null | 通常否 | 由脚本根据最终跳转 URL、canonical link、去参数规则生成 | articles | 主要是格式规范化与去重 |
| `published_at` | datetime/null | 可能有 | 从 RSS `published` / `pubDate` / `updated` 解析；缺失则置空，不用抓取时间冒充发布时间 | articles | 需要格式转换为标准 datetime |
| `fetched_at` | datetime | 否 | 抓取脚本运行时写入 | articles | 系统时间，不来自 RSS |
| `source_key` | string | 否 | 由清洗脚本按 `source_name`、抓取入口或 raw 文件绑定到 `source_profiles.source_key` | articles | 稳定内部来源 ID，不依赖 RSS 文本 |
| `source_name` | string | 可能有 | 优先取 `source_profiles.source_name`；RSS feed title/source/公众号名只作匹配线索 | articles（快照） | 只要命中 `source_name/source_key`，即可自动带出来源分类 |
| `source` | enum | 否 | 从 `source_profiles.source` 复制到文章快照 | articles | 合并表达渠道 + 来源性质，如 `wechat_official`、`wechat_domestic_media`、`web_official` |
| `source_list` | string[] | 否 | 从 `source_profiles.source_list` 复制到文章快照 | articles | 记录该来源来自客户 Excel 哪类清单，便于 QA 和后续配置维护 |
| `ingest_method` | enum | 否 | 从 `source_profiles.ingest_method` 或抓取入口复制 | articles 或 import_runs | 技术抓取方式，不由 RSS 自带 |
| `content_language` | enum | 通常否 | 优先用 `source_profiles.language` 默认值；必要时由脚本对标题/摘要/正文检测修正 | articles | RSS language 只能作为辅助信号 |
| `market_region` | tag[] | 通常否 | 优先用 `source_profiles.market_region` 默认值；必要时结合 URL、来源、标题/正文显式地名补充 | articles | 来源配置为主，内容规则为辅 |
| `extraction_status` | enum | 否 | 正文抽取脚本根据正文质量写入 `success` / `partial` | article_contents | 正文抽取结果，不来自 RSS |
| `raw_tags` | string[] | 可能有 | 读取 RSS category/tags 或原文标签，去重、去空、统一为数组 | article_contents | 只做原文标签保留，不等同于本字典 taxonomy |

> 说明：`source`、`source_list`、`ingest_method`、默认语言和默认地区等来自媒体源配置，记录在 `source_profiles` / `articles`，**不写进 `article_tags`（标签-文章关联表）**。打标 Agent 只读取这些字段作为上下文。

### 1.2 Agent 判断标签（tags，OpenClaw 语义判断）

| 字段 | 类型 | 用途 | 约束 |
|------|------|------|------|
| `primary_story_type` | tag[] | 文章在报道哪类事件/内容 | **多选**；至少 1 个 |
| `product_application` | tag[] | 涉及的产品品类 / 应用场景 | 多选；字典外用 `other:<名>` |
| `ingredient_technology` | tag[] | 涉及的成分或核心技术（主轴维度） | 多选；字典外用 `other:<名>` |
| `functional_claim` | tag[] | 宣称的功效 / 消费者利益 | 多选；字典外用 `other:<名>` |
| `value_chain_stage` | enum | 文章主要影响产业链哪个位置 | 单选；跨链用 `cross_chain` |
| `company` | string[] | 抽取文章事件涉及的公司实体 | 自由文本；后端与 Watchlist 归一化匹配 |

> 本版已删除：`industry_segment`（并入 `value_chain_stage`）、`strategic_driver`、`entity_role`（公司角色改由 Watchlist 配置承载，见第六节）。

### 1.3 多维协同范式

#### 核心原则：多维交叉描述，而非逐级筛选

本标签体系不是"漏斗式"层层过滤，而是**多个独立切面同时描述一篇文章**。每个维度只回答一个切面问题，不越界、不重复。

#### 七个切面各司其职

| 维度 | 回答的问题 | 不回答的问题 |
|------|-----------|-------------|
| `relevance` | 这篇文章是否值得进入体系？ | 文章具体讲了什么 |
| `primary_story_type` | 发生了哪类事件？ | 发生在产业链的哪一段、用的什么技术 |
| `value_chain_stage` | 核心事件的主舞台在产业链哪一段？ | 事件类型、具体技术 |
| `ingredient_technology` | 涉及什么成分/技术？ | 事件类型、产品场景 |
| `product_application` | 涉及什么产品品类？ | 功效、技术原理 |
| `functional_claim` | 宣称什么功效？ | 成分机制、产品形态 |
| `company` | 涉及哪些企业？ | 企业的角色/属性（由 Watchlist 承载） |

#### 使用范式：组合查询，而非单维度依赖

用户或下游 newsletter 生成不是"找某个标签"，而是**组合多个维度精确定位**：

> 不是"找产品发布" -> 而是"找**原料端**的**产品发布**，涉及**多肽**，用于**护肤**，功效**抗衰**"。

一篇文章由多个维度的**交集**共同定义。`primary_story_type` 只回答"是产品发布"，不负责说明这是原料发布还是品牌发布；那是 `value_chain_stage` 的事。它也不回答发布的是什么；那是 `ingredient_technology` 和 `product_application` 的事。

#### 边界说明：避免维度大包大揽

- 不因为 `value_chain_stage=brand_finished_product` 就判定文章不相关；准入由 `relevance` 单独判断。
- 不因为 `primary_story_type=corporate_move` 就继续拆分公司动作子类型；召回、并购、融资等细节由 `evidence_text` 和 `company` 承载。
- 不因为一篇文章讲的是欧莱雅面霜，就要求 `value_chain_stage` 必须体现上游原料；原料和技术信息由 `ingredient_technology` 和 `company` 承载。

> 一句话概括：**每个维度只回答一个切面，组合起来才是一篇文章的完整画像。**

## 二、推荐标准输出对象

### 2.1 `relevance` 准入标准

`relevance` 是体系的大门，不是普通标签。输出值沿用 schema：`relevant` / `not_relevant` / `unclear`。其中 `not_relevant` 对应业务讨论中的 "irrelevant"。

| 输出值 | 准入标准 |
|--------|----------|
| `relevant` | 文章涉及美妆个护原料、成分、配方、功效、技术创新、政策法规，对禾大情报有价值。下游品牌新闻须涉及成分、配方、原料采购或功效，才算相关。 |
| `not_relevant` | 纯品牌明星代言、纯渠道销售数据（不涉及成分）、纯企业内部行政变动、纯 C 端促销。 |
| `unclear` | 正文不足、证据不完整、标题/摘要无法判断是否与美妆个护原料 intelligence 相关。 |

### 2.2 输出示例

## 三、基础数据字段词表

### 3.1 抓取方式 `ingest_method`

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `wechat_account` | 微信公众号 | 公众号文章抓取（主力） |
| `direct_rss` | 直接 RSS | 媒体官网或官方 RSS |
| `google_news_rss` | Google News RSS | 替代源 |
| `web_scraper` | 官网/网页抓取 | 竞品官网、展会官网新闻稿 |
| `manual_or_reference` | 手动/参考补充 | 人工整理、转载页 |
| `other` | 其他 | 无法归类 |

> 已删除 `database_query`：当前未接入 SpecialChem、Research and Markets 等专业数据库订阅，设计上不考虑中间数据库查询。
> 已删除 `unavailable`、`access_status` 及其失败枚举（`blocked_waf`、`blocked_401`、`no_rss` 等）：抓取失败的文章不入库，失败信息留在抓取日志（`article_import_runs.error_log`）。

### 3.2 抽取状态 `extraction_status`

只保留"已入库文章可能出现"的状态：

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `success` | 成功 | 正文可用 |
| `partial` | 部分成功 | 有正文但含导航/广告或缺失 |

> `failed` / `blocked` / `not_attempted` / `link_dead` 不在此枚举：抓取/抽取失败者不入库；原文链接有效性是**月报生成阶段**的检查（记录在 `articles.link_status`），不是入库标签。

### 3.3 语言 `content_language`

`zh`（主力）、`en`、`ja`、`ko`、`de`、`fr`、`mixed`、`unknown`、`other`。

### 3.4 地区 `market_region`

仅在文章或来源明确指向地区时写入：`china`、`global`、`europe`、`north_america`、`japan_korea`、`sea`、`middle_east`、`other`。

### 3.5 来源分类 `source`

`source` 来自 `source_profiles` 媒体源配置，不由 Agent 判断。它用一个值同时表达“渠道 + 来源性质”：例如“禾大自己的公众号”在 media list 中配置为 `source_name=禾大...`、`source=wechat_official`，清洗脚本只需命中 `source_name/source_key`，即可自动带出 `source`。

RSS 原始 JSON 可能带有 feed title、source 或 category，但通常不会稳定自带业务来源分类。旧的独立 `source_nature` 不再作为文章基础字段保留，其含义已并入 `source`。

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `web_official` | 官网/官方网页 | 公司官网、新闻中心、监管机构、协会、展会官网等官方网页 |
| `wechat_official` | 官方微信公众号 | 企业、品牌、机构或展会的官方公众号 |
| `wechat_domestic_media` | 微信国内媒体 | 国内行业媒体公众号，如青眼、聚美丽、美妆小报 |
| `wechat_event_official` | 展会/活动官方微信 | PCHi 等展会或活动官方公众号 |
| `wechat_third_party_analysis` | 微信第三方分析/数据平台 | 美丽修行 App 等数据平台、分析机构公众号 |
| `rss_media` | RSS 媒体源 | 海外行业媒体 RSS / Atom 来源 |
| `google_news_media` | Google News 媒体源 | Google News RSS 替代源 |
| `manual_reference` | 人工/参考补充 | 人工整理、客户补充、无稳定自动抓取入口的来源 |

## 四、Agent 判断标签词表

### 4.1 主故事类型 `primary_story_type`（多选，≥1）

回答"这篇文章在报道哪类事情"。一篇文章可同时属于多类（如"新品发布"+"技术创新"）。

| 标签值 | 中文名 | 定义 | 备注 |
|--------|--------|------|------|
| `corporate_move` | 企业动态 | 公司层面的动作：并购、合作、产能/建厂、出海、投融资、IPO/上市、估值、组织调整、市场进入、经销协议，以及公司侧的产品召回/质量处理 | 合并自旧 `business_transaction` + `capital_finance`，并吸收供应链中"出海建厂"等公司动作 |
| `product_launch_or_update` | 新原料/新品发布 | 新原料、新活性物、新配方方案、商业化发布或升级 | |
| `technology_process_innovation` | 技术/工艺创新 | 递送系统、合成生物、发酵、AI 研发、绿色化学、加工技术突破 | |
| `research_science` | 科研/功效验证 | 学术研究、临床/功效测试、专利、机理验证、科学证据 | |
| `regulation_policy` | 政策法规 | 新原料备案、法规、标准、认证、标签、致敏原、监管动作；以及**监管侧**的召回令、风险评估、限量、检测标准 | 吸收旧 `safety_quality_issue` 的监管/安全部分 |
| `market_consumer_insight` | 市场/消费者洞察 | 市场规模、品类走势、成分热度、消费者偏好、趋势报告；以及供应链上下游的**原料渠道、价格、出海/进出口的市场趋势** | 吸收旧 `supply_chain_trade` 的市场/趋势部分 |
| `event_news` | 活动/会议 | 展会、峰会、论坛、研讨会预告或会后报道 | PCHi、in-cosmetics、ICIC、NYSCC |
| `other` | 其他 | 相关但无法归入 | 进复核 |

> 已删除：`business_transaction`、`capital_finance`（→ `corporate_move`）、`safety_quality_issue`（拆入 `regulation_policy` / `corporate_move`）、`supply_chain_trade`（拆入 `market_consumer_insight` / `corporate_move`）。
> `technology_process_innovation` 与 `research_science` 保留为合并后的两个高层标签：前者看技术/工艺突破，后者看科学证据和验证。

### 4.2 产品/应用场景 `product_application`（多选，可空，活字典）

仅在文章明确涉及产品品类/应用场景时填写。字典无法判断场景时，用 `other:<名>`（活字典，见第五节），不要勉强套用现有值。

### 4.6 公司 `company`（自由实体 + Watchlist 匹配）

`company` 是 Agent 从文章中抽取的公司实体显示名，不是字典标签。

**写入规则（仅当公司是事件主体或重要相关方）：** 公司发生新品/新原料、并购、融资上市、合作、产能、备案、研发、出海等动作；或文章围绕该公司高管/品牌/子公司展开；或该公司是事件主要合作方；或第三方分析的重点案例。

**不写入：** 仅引用其员工一句话但事件与该公司无关；只在广告/导航/相关阅读/长名单中出现；只是行业背景或历史例子；仅在趋势分析中作为"某品牌也在用"的案例引用出现，即使提到名字但不是分析重点。

**与 Watchlist 的关系（重要）：** 公司**角色**（竞品 / 客户 / 代理 / 生态）不再由 Agent 打标（`entity_role` 已删除），而是由后端用抽取出的 `company` 去匹配**客户配置的 Watchlist**（见第六节）得出。Agent 只负责抽取公司显示名。

```json
{ "company": ["Croda", "维琪"], "company_normalized": ["croda", "winkey"] }
```

## 五、活字典（Picklist）与维护

活字典有两条路径：

| 路径 | 适用字段 | 机制 | 实时性 |
|------|----------|------|--------|
| **A. 内联 `other:` 实时抽取** | `ingredient_technology`、`product_application`、`functional_claim`（开放词表） | Agent 实时打 `other:<名>` + `extracted_name` + 证据；频次驱动晋升 | 实时，文章当下可检索 |
| **B. `suggested_new_tags` 建议** | 封闭枚举字段（`primary_story_type`、`value_chain_stage`） | Agent 提结构化建议进复核队列，人工 qualification | 非实时 |

### 5.A 路径 A：开放字段的内联 `other:`（活字典核心）

> **核心原则：字典里没有对应的成分/应用/功效，宁可多写一个 AI 抽取出来的名字，也不要因未收录就不打标，从而错过新出现的趋势。不要把识别工作全部留到人工审核——人工审核非实时、且需重读文章，成本太高。**

1. **数据标注与写入（实时，Agent 完成）**
   - 字典外的值，直接在该字段写 `"other:<slug>"`（如 `other:exosome_lipid_nanoparticle`、`other:cica_centella`）。
   - `slug` 归一化：全小写，空格和连字符转下划线，去掉冠词（`a` / `an` / `the`），单数化（去掉末尾 `s`）。
   - 为每个 `other:` 标签附 evidence_record，并在 `extracted_name` 写出原文显示名。
   - 文章**照常 `tagged`**，不因 `other:` 转 `needs_review`。
2. **字典更新机制（频次驱动，人工不重读文章）**
   - 后端对 `other:<slug>` 归一化后做**频次统计**（落在 `inline_other_terms` 表与 `v_inline_other_frequency` 视图）。
   - 某 `other:X` 在窗口期内出现 **≥2 篇不同文章**或 **≥2 个不同来源**时，进入人工**批量**审核（只看高频列表 + 抽样证据）。
   - 通过则把该值写入**字典表 `taxonomy_terms`**（`status=active`），成为正式 picklist 项，并可重打标历史 `other:`。
3. **如何创建一个 picklist（活字典落库）**
   - 每个标签字段的正式可选值 = `SELECT label FROM taxonomy_terms WHERE field=? AND status='active'`。
   - 初始 picklist 由本字典词表**种子导入**（`origin='seed'`）。
   - 活字典晋升把 `inline_other_terms` / `suggested_new_tags` 中通过的项写入 `taxonomy_terms`（`origin='promoted_*'`）。
   - 详见 `dev/docs/数据库设计.md` 的 `taxonomy_terms` 表。

### 5.B 路径 B：封闭字段的 `suggested_new_tags`

当封闭枚举字段（`primary_story_type`、`value_chain_stage`）确实缺少标签时，Agent 返回 `suggested_new_tags`（结构：`field/label/display_name/parent_label/reason/evidence/status=pending_review`），进复核队列。`ingredient_technology` / `product_application` / `functional_claim` **不走此路**，改走 5.A 内联 `other:`。

### 5.C 人工审核规则（两路通用，至少满足两条）
- 窗口期内 ≥2 篇不同文章或 ≥2 个不同来源。
- 不能被现有标签准确表达。
- 显著提升 newsletter 筛选、栏目归类或客户匹配。
- 定义边界清晰，Agent 可稳定复现。
- 不是一次性公司名、活动名、单一产品名或过窄事件。

> 成分迭代快，`ingredient_technology` 的 `other:` 频次审核建议**每两周**一次。

## 六、Watchlist（客户/竞品监测名单）设计

Watchlist 是**客户配置**，用于把抽取出的 `company` 归一化并标注角色，回答"竞品/客户最近在做什么"。它不是 taxonomy，不进 `suggested_new_tags`，不回写本字典词表。

### 6.1 来源
- Excel 名单中的**客户相关公司**（MNC 品牌、国内品牌）。
- **竞品公司**（国际/国内原料商）。
- 代理商、行业媒体/机构、禾大自身。

### 6.2 字段（落库于 `watchlist_entities`，详见数据库设计）

| 字段 | 说明 |
|------|------|
| `display_name` | 规范显示名 |
| `aliases` | 别名/中英文名，**规范化**：如统一"欧莱雅"="L'Oréal"、"禾大"="Croda" |
| `entity_role` | 角色（配置项，非标签）：`self` / `competitor` / `customer` / `channel_partner` / `ecosystem` |
| `market_region` | `china` / `global` / `europe` …（用于还原"国际/国内"） |
| `value_chain_stage` | 可选，公司主营链位置 |
| `status` | `active` / `paused` / `archived` |

### 6.3 维护与展示
- **规范化**：别名合并，确保同一公司不同写法归一。
- **存储逻辑**：公司名不进 `suggested_new_tags`、不回写字典词表；只在 Watchlist 配置中维护。
- **维护方式**：由客户配置并独立维护，定期与客户同步增删（新增竞品、新锐原料商等）。
- **展示形式**：在本字典中作为独立一节（即本节）维护清单来源与规范，落库为 `watchlist_entities`；后端通过 `v_watchlist_matches` 把 `company` 与 Watchlist 匹配出角色。

## 七、AI 打标约束

### 7.1 推荐判断顺序
1. 沿用入库脚本写好的字段（来源、语言、地区、来源性质等），不重算。
2. 判断文章是否与**美妆个护原料 intelligence** 相关（下游品牌新闻须涉及成分/配方/原料采购/功效才算相关）。
3. 判断 `primary_story_type`（多选，≥1）。
4. 有明确品类时填 `product_application`（字典外用 `other:`）。
5. 有明确成分/技术时填 `ingredient_technology`（字典外用 `other:`，宁多勿漏）。
6. 有明确功效逻辑时填 `functional_claim`（字典外用 `other:`）。
7. 单选 `value_chain_stage`。
8. 抽取公司到 `company`。
9. 为每个标签（含每个 `other:`）提供一条 `evidence_text`；`other:` 另给 `extracted_name`。
10. 封闭字段缺标签时写 `suggested_new_tags`，不创造正式标签。

### 7.2 防重复与边界
- 不为凑数量补标签。**例外**：`ingredient_technology` / `product_application` / `functional_claim` 的字典外值宁可多打 `other:`。
- 公司名、品牌名、展会名不进 taxonomy；公司只进 `company`。
- 角色判断不在打标阶段做（无 `entity_role`），由 Watchlist 匹配得出。
- 不新增 `corporate_move` 子维度；企业动作细节保留在证据和报告叙事中。
- 不增加文章体裁字段；文章类型以现有 `primary_story_type` 与证据承载。

### 7.3 证据规则（已精简）
- 每个标签必须有一条**客观证据原文 `evidence_text`**（取自标题/摘要/正文）。
- 不再要求 `evidence_field` 与 `reason`。
- `other:<名>` 必须额外带 `extracted_name`。
