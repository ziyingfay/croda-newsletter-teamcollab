# 标签字段字典 · 禾大 Croda 美妆个护原料版

> 本字典是为客户**禾大 Croda Beauty**（美妆个护原料商）定制的 newsletter 文章打标词表。它**完全继承**本项目两层 MECE 架构（需求变更 009）、证据驱动、活字典、公司自由实体、无 confidence 等原则，仅将 Agent 判断字段的词表从"食品饮料"重构为"美妆个护原料"行业。
>
> 配套需求分析见同目录 `禾大Croda-Newsletter需求分析报告.md`。
> 父项目食品饮料字典已归档到 `reference/legacy-food-beverage/标签字段字典-食品饮料版.md`。
>
> schema 命名建议：`newsletter-tagging/croda-beauty-v1`

---

## 设计原则（继承自父项目）

1. **两层处理**：基础数据字段由代码/轻量规则处理；深度标签由 Agent 阅读文章后判断。
2. **MECE 优先**：每个字段一个独立维度，相同含义不得跨字段重复。
3. **客户匹配导向**：字段必须帮后端把文章精准匹配到禾大关注的竞品、客户品牌、成分、技术、功效议题或战略主题。
4. **不盲目细分**：不能稳定提升筛选价值的标签不进字典；新成分/技术先进候选池。
5. **公司不入字典**：公司名由 Agent 自由抽取写入 `company`；监测名单（竞品/客户/媒体 Excel）是**客户配置**，不是 taxonomy。`entity_role` 是受控的角色维度，可入字典。
6. **证据驱动**：Agent 判断字段必须有标题、摘要、正文、原文标签或 URL 证据。
7. **无 confidence**：不使用 AI 自评 confidence；质量靠 schema validation、证据校验、post-processing、review queue 和人工抽样。

## 一、两层字段结构

### 1.1 基础数据字段标签（代码 / 轻量规则）

与父项目一致，仅 `market_region` / `content_language` 取值贴合美妆个护以中国为核心、覆盖欧美日韩的实际。

| 字段 | 类型 | 处理方式 | 用途 |
|------|------|----------|------|
| `source_key` | string | 来源配置 / 入库脚本 | 稳定来源 ID |
| `source_name` | string | 来源配置 / RSS / 公众号名 | 来源展示名 |
| `url` / `canonical_url` | string | 入库脚本 | 去重、追溯、原文链接（月报强制有效链接） |
| `published_at` | datetime/null | RSS / 解析脚本 | 30 天回溯时间筛选 |
| `fetched_at` | datetime | 抓取脚本 | 抓取批次追踪 |
| `access_status` | enum | 来源配置 | RSS/公众号/官网可用性 |
| `ingest_method` | enum | 来源配置 / URL | 公众号抓取、直接 RSS、Google News、官网 scraper 等 |
| `extraction_status` | enum | 正文抽取脚本 | 正文质量与是否复核；支撑"链接失效"标注 |
| `content_language` | enum | 语言检测 | 语言筛选（zh 为主、en 为辅，支撑双语切换） |
| `market_region` | tag[] | 来源配置 / 地名扫描 / URL | 地区筛选；仅显式证据时写入 |
| `source_nature` | enum | 来源配置 / URL / 署名 | 信息来源性质（公众号占比高） |
| `raw_tags` | string[] | 原文标签 / RSS | 原始标签保存，不直接等于正式标签 |

### 1.2 Agent 判断标签（LLM 语义）

| 字段 | 类型 | 用途 | 约束 |
|------|------|------|------|
| `primary_story_type` | enum | 文章主要是什么类型的事件/内容 | 单选，必填一个主类型 |
| `industry_segment` | enum | 文章主要属于美妆个护产业链哪个大段 | 单选；跨段用 `cross_sector` |
| `product_application` | tag[] | 涉及的产品品类 / 应用场景 | 多选，仅明确品类时填写 |
| `ingredient_technology` | tag[] | **涉及的成分或核心技术**（本客户主轴维度） | 多选；受控 + 字典外用内联 `other:<名>` 实时打标（见 4.4.1） |
| `functional_claim` | tag[] | **宣称的功效 / 消费者利益** | 多选，建议 0-3 个；仅有明确功效逻辑时填 |
| `strategic_driver` | tag[] | 文章为什么对禾大有业务意义 | 多选，建议 1-3 个 |
| `value_chain_stage` | enum | 主要影响产业链哪个位置（仅链位置，不含公司关系） | 单选；跨链用 `cross_chain` |
| `entity_role` | tag[] | 文章主体公司**相对禾大的业务关系**（仅关系，不含链位置） | 多选；受控词表 |
| `company` | string[] | 抽取文章事件涉及的公司实体 | 自由文本，不受字典限制 |
| `suggested_new_tags` | object[] | 现有 taxonomy 不足时提新标签建议 | 进复核队列；对 `ingredient_technology` 降为可选（见 4.4.1） |

## 三、基础数据字段词表

### 3.1 来源状态 `access_status`

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `working` | 正常 RSS | 可直接通过 RSS 获取 |
| `wechat_ok` | 公众号可抓 | 通过公众号抓取链路可获取（主要来源） |
| `google_news` | Google News RSS | 官网/公众号不可用但可通过 Google News 获取 |
| `web_scrape` | 官网抓取 | 通过竞品官网 scraper 获取新闻稿 |
| `blocked_waf` | WAF 封锁 | 403、CloudFlare、企业级 WAF 阻断 |
| `blocked_401` | 认证阻断 | 需登录、订阅或认证（如部分数据库） |
| `no_rss` | 无可用源 | 未找到可用抓取链路 |
| `other` | 其他 | 新抓取状态，需人工确认 |

### 3.2 抓取方式 `ingest_method`

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `wechat_account` | 微信公众号 | 公众号文章抓取（主力） |
| `direct_rss` | 直接 RSS | 媒体官网或官方 RSS |
| `google_news_rss` | Google News RSS | 替代源 |
| `web_scraper` | 官网/网页抓取 | 竞品官网、展会官网新闻稿 |
| `database_query` | 数据库查询 | SpecialChem、Research and Markets 等专业数据库 |
| `manual_or_reference` | 手动/参考补充 | 人工整理、转载页、旧样本 |
| `unavailable` | 暂不可抓取 | 无源、认证、WAF 阻断 |
| `other` | 其他 | 无法归类 |

| 标签值 | 中文名 | 定义 |
|--------|--------|------|

#### 4.4.1 字典外成分/技术的实时 Others 抽取（活字典核心机制）

> **核心原则：宁可多打一个 AI 抽取出的成分/技术名，也不要因字典未收录而漏标，从而错过新出现的成分或技术。**

美妆成分/技术迭代极快，等人工把新成分加进字典再打标会漏掉窗口期内的新趋势，而人工审核不是实时的、且需重读文章，成本高。因此**字典外成分/技术不走"先丢 suggestion、等人工"的老路，而是由 Agent 实时内联打标**：

**1）数据标注与写入（实时，由 Agent 完成）**
- 当文章出现 `ingredient_technology` 词表未收录的具体成分或技术时，**不得跳过、不得只填 `suggested_new_tags`**。
- 直接在 `ingredient_technology` 内写入 `"other:<slug>"`，例如 `"other:snow_algae_extract"`、`"other:synthetic_pdrn"`、`"other:exosome_lipid_nanoparticle"`。
- 为每个 `other:` 标签附 evidence_record，并在 `extracted_name` 写出原文显示名（如"雪藻提取物""人工合成 PDRN"），`evidence_text` 引用原文证据。
- 只要文章确实涉及某成分/技术就打，哪怕只出现一次——**多打优于漏打**。
- 文章**不因出现字典外成分而被打成 `needs_review`**；它照常 `tagged`，只是带 `other:` 标签。这样该新成分**当下即可被检索、聚类、计数**。

```json
"ingredient_technology": ["peptides", "other:exosome_lipid_nanoparticle"],
"evidence_records": [
  {
    "field": "ingredient_technology",
    "label": "other:exosome_lipid_nanoparticle",
    "extracted_name": "外泌体脂质纳米粒",
    "evidence_field": "content",
    "evidence_text": "该公司推出外泌体脂质纳米粒递送平台，用于活性物靶向递送。",
    "reason": "字典暂无此成分/技术，实时内联抽取，避免漏标。"
  }
]
```

**2）字典更新机制（频次驱动，人工只看高频列表，不重读文章）**
- 后端对 `other:<slug>` 做归一化（合并 `snow_algae` / `snow-algae-extract` 等写法）后**统计频次**。
- 当同一新成分/技术在窗口期内出现 **≥2 篇不同文章** 或 **≥2 个不同来源**，进入人工**批量**审核：审核员只看高频 `other:` 列表与计数 + 抽样证据，**无需重读每篇文章**。
- 通过后正式加入 `ingredient_technology` 词表，并可触发历史 `other:<slug>` → 正式标签的归一化重打标。
- 一次性、过窄或写错的 `other:` 因频次不足自然沉淀，不污染正式词表。

**3）与 `suggested_new_tags` 的分工**
- 对 `ingredient_technology`，**内联 `other:` 是强制且实时的主机制**；`suggested_new_tags` 降为可选，仅当 Agent 想额外建议正式标签名/父类时填，且**不再是打标前置条件**。
- 其它**封闭枚举字段**（`primary_story_type`、`industry_segment`、`product_application`、`value_chain_stage`、`entity_role`、`source_nature`）仍严格受控，不用内联 `other:` 自由扩展；无法归类时填其 `other` 值并进复核。`functional_claim` 同理保持受控。

### 4.6 战略驱动 `strategic_driver`（多选，建议 1-3）

| 标签值 | 中文名 | 定义 | 排除条件 |

| `delivery_technology` | 递送技术 | 以递送/缓释/靶向作为核心差异化 | 普通配方不用 |
| `biotech_synthetic_biology` | 生物技术/合成生物 | 合成生物、发酵、生物制造作为战略主题 | 只提一句不用 |
| `ai_digital_rd` | AI/数字化研发 | AI、数字化驱动研发或供应链且是核心 | AI 路人式提及不用 |
| `sustainability` | 可持续 | 碳足迹、RSPO、再生、生物质平衡、环保包装、循环 | 仅 vague green/eco/natural 不用 |
| `clean_natural_safety` | 纯净/天然/安全 | 纯净美妆、天然来源、温和、无添加、安全性 | 与可持续不可自动混淆 |

| `localization_substitution` | 国产替代 | 国产原料替代进口 | 普通国内新闻不用 |

| `other` | 其他 | 明确业务驱动但未列入 |

| 标签值 | 中文名 | 定义 |
|--------|--------|------|
| `raw_material_upstream` | 上游原料/生物制造 | 发酵、合成、菌株、上游原材料供应 |
| `ingredient_active` | 原料/活性成分 | 活性物与原料的研发、生产、商业化 |
| `formulation_application` | 配方/应用 | 配方开发、应用方案、复配、感官调节 |
| `contract_manufacturing` | 代工/制造 | 代工厂、生产工艺、设备、产能 |

| `other` | 其他 | 明确位置但未列入 |

### 4.8 实体关系 `entity_role`（多选，受控）

> **去重说明（与 `value_chain_stage` 解耦）：** 本字段**只表达公司相对禾大的业务关系**，不再编码产业链位置。链位置交给 `value_chain_stage`，产业大段交给 `industry_segment`，"国际/国内"交给 `market_region`，具体公司交给 `company`。这样 `entity_role` 与 `value_chain_stage` 不再有重复语义：前者回答"这家公司是禾大的谁"，后者回答"这件事发生在链条哪一段"。
>
> 公司**名称**仍是 `company` 的自由实体抽取（见 4.9）；`entity_role` 是受控的关系维度。一篇文章可涉及多个关系（如竞品向某客户品牌供货 → `competitor` + `customer`）。关系判断以**该公司在本文中相对禾大的身份**为准。

| 标签值 | 中文名 | 定义 | 链位置在哪表达 |
|--------|--------|------|----------------|
| `self` | 禾大自身 | Croda/禾大及其品牌产品（Matrixyl 等） | value_chain_stage |
| `competitor` | 竞品 | 与禾大构成竞争的原料/活性物供应商（国际国内不限，由 market_region 区分） | 通常 `ingredient_active`，但不绑定 |
| `customer` | 客户 | 下游原料采购方：品牌方、代工厂等（MNC vs 国货由 market_region + company 区分） | `brand_finished_product` 或 `contract_manufacturing` |
| `channel_partner` | 渠道伙伴 | 代理商、经销商、分销合作方 | `distribution_channel` |
| `ecosystem` | 行业生态方 | 行业媒体、公众号、科研院校、协会、监管机构、数据库等中立第三方 | `regulation_research` 或不适用 |
| `other` | 其他 | 关系明确但未列入 | — |

**旧值迁移对照（供历史数据归一化）：** `competitor_ingredient`→`competitor`；`customer_brand_mnc`/`customer_brand_domestic`/`cmo_oem`→`customer`（地理与代工身份分别由 `market_region`、`value_chain_stage` 表达）；`distributor_agent`→`channel_partner`；`industry_media`/`academic_institution`/`regulator_association`→`ecosystem`。

> 月报仍可还原"国际/国内原料商""竞品矩阵""客户品牌信号"等分类：用 `entity_role` × `market_region`（competitor × europe/na = 国际原料商；competitor × china = 国内原料商）+ `company` 即可，无需在 `entity_role` 内重复编码地理与位置。

### 4.9 公司 `company`（自由实体，不入字典词表）

`company` 是 Agent 从文章中抽取的公司实体显示名，不是字典标签。用于回答"某家公司最近在做什么"，并支撑禾大对竞品/客户/代理/媒体的按名筛选。

**写入规则（仅当公司是事件主体或重要相关方）：**
- 标题、摘要或正文核心段落直接讨论该公司。
- 公司发生新品/新原料、并购、融资上市、合作、产能、备案、研发、出海、可持续等动作。
- 文章围绕该公司高管、品牌、子公司或明确归属业务展开。
- 第三方分析中该公司是被重点分析的案例。

**不写入：**
- 仅引用其员工一句话但事件与该公司无关。
- 只在广告、导航、相关阅读、赞助/长名单中出现。
- 公司名只是行业背景或历史例子。
- 下游客户品牌新闻**与成分/配方/原料采购/功效无关**（如纯代言、营销活动）——此时品牌不作为 `company` 主体写入，必要时整篇判低相关。

**存储建议：** 保留显示名（`Croda`、`BASF`、`维琪`），入库脚本另生成标准化键并与**客户监测名单（Excel watchlist 配置）**匹配，标注 `entity_role`。

```json
{
  "company": ["Croda", "维琪"],
  "company_normalized": ["croda", "winkey"],
  "entity_role": ["self", "competitor_ingredient"]
}
```

公司名不进入 `suggested_new_tags`，不回写本字典词表。watchlist 是客户配置，独立维护、定期与客户同步增删。

## 五、活字典维护：两条路径

本字典的"活字典"有两条互补路径，按字段类型分工：

| 路径 | 适用字段 | 机制 | 实时性 |
|------|----------|------|--------|
| **A. 内联 `other:` 实时抽取** | `ingredient_technology`（开放词表，主轴） | Agent 实时打 `other:<名>`，频次驱动晋升（见 4.4.1） | 实时，文章当下可检索 |
| **B. `suggested_new_tags` 建议** | 其它 taxonomy 字段（strategic_driver、functional_claim 等封闭词表） | Agent 提结构化建议进复核队列，人工 qualification | 非实时 |

> **为什么分两条：** 成分/技术迭代最快、漏标代价最高，所以走实时内联 A，宁可多打也不漏；其它维度相对稳定、且自由扩张风险高，仍走受控的 B。**不要把成分/技术的识别全部留到 B（人工审核非实时、需重读文章，成本过高）。**

### 5.A 路径 A：`ingredient_technology` 内联 `other:`
见 4.4.1。要点：Agent 实时写 `other:<slug>` + `extracted_name` + 证据 → 后端按归一化频次统计 → 高频（窗口期 ≥2 篇/≥2 来源）进人工**批量**审核（只看高频列表，不重读文章）→ 晋升为正式标签并可重打标历史 `other:`。

### 5.B 路径 B：其它字段的 `suggested_new_tags`

当文章相关、但封闭词表字段（如 `strategic_driver`、`functional_claim`）无法准确表达其长期筛选价值时，Agent 返回 `suggested_new_tags`（不适用于公司名，也不再要求 `ingredient_technology` 走此路）。

### 5.1 使用条件
1. 内容明确属于某现有字段的判断范围。
2. 正式词表无合适标签。
3. 新标签对 newsletter 筛选/栏目/客户匹配有长期价值。
4. 能提供证据。

证据不足、正文缺失或语义模糊时不要建议新标签，应进 `needs_review`。

### 5.2 候选结构（路径 B 示例）

```json
{
  "field": "strategic_driver",
  "label": "longevity_economy",
  "display_name": "长寿经济",
  "parent_label": "market_expansion",
  "reason": "多篇文章把抗衰原料置于'长寿/健康老龄'消费趋势下，现有 strategic_driver 只能泛化表达。",
  "evidence": "title/content",
  "status": "pending_review"
}
```

> 注：`ingredient_technology` 的新成分/技术**不走本结构**，改走 4.4.1 的内联 `other:`。

### 5.3 人工审核规则（两条路径通用，至少满足两条才入库）
- 窗口期内出现不少于 2 篇不同文章，或来自不少于 2 个不同来源。
- 不能被现有标签准确表达。
- 显著提升 newsletter 筛选、栏目归类或客户匹配。
- 定义边界清晰，Agent 可稳定复现。
- 不是一次性公司名、活动名、单一产品名或过窄事件。

> 美妆成分迭代快，`ingredient_technology` 的 `other:` 频次审核节奏建议**每两周**（高于食品饮料版每周），以快速纳入外泌体、合成 PDRN、蓝生物技术等新兴成分。

## 六、AI 打标约束

### 6.1 推荐判断顺序
1. 读取基础字段，不重复推理代码已确定字段（除非明显冲突）。
2. 判断文章是否与**美妆个护原料 intelligence** 相关（下游品牌新闻须与成分/配方/原料/功效相关才算相关）。
3. 判断 `source_nature`。
4. 单选 `primary_story_type`。
5. 单选 `industry_segment`。
6. 有明确品类时填 `product_application`。
7. **有明确成分/技术时填 `ingredient_technology`（本客户重点）；字典外的实时写 `other:<名>`，不漏标（见 4.4.1）**。
8. 有明确功效逻辑时填 `functional_claim`。
9. 判断 1-3 个 `strategic_driver`。
10. 单选 `value_chain_stage`（只判链位置）。
11. 抽取公司到 `company`，并判断 `entity_role`（只判与禾大的关系：self/competitor/customer/channel_partner/ecosystem）。
12. 为每个 Agent 判断字段（含每个 `other:` 标签）提供 evidence record。
13. 封闭词表字段缺失长期有价值标签时写 `suggested_new_tags`，不创造正式标签；`ingredient_technology` 改走内联 `other:`。

### 6.2 防重复规则
- `ingredient_technology`（成分/技术是什么）与 `functional_claim`（宣称什么功效）与 `strategic_driver`（对禾大为何重要）是三个独立维度，不可互相替代。例：多肽(ingredient) + 抗衰(claim) + 递送技术(driver) 可共存。
- 不把 `primary_story_type=market_consumer_insight` 与 `strategic_driver=market_expansion` 当同一层意思。
- `industry_segment=packaging_delivery` 与 `value_chain_stage`/`ingredient_technology=encapsulation_delivery` 表示不同维度；仅当文章确属包装大段且影响该阶段才共存。
- `sustainability` 只能是 `strategic_driver`，不能当故事类型。
- **`entity_role` 与 `value_chain_stage` 不可互相替代**：前者只表达"公司相对禾大的关系"，后者只表达"链位置"。竞品/客户/渠道伙伴的身份进 `entity_role`，"原料/品牌/渠道/代工"等位置进 `value_chain_stage`，"国际/国内"进 `market_region`。不要在 `entity_role` 里编码位置或地理。
- 公司名、品牌名、展会名不放入 taxonomy 标签；公司名只进 `company`，关系进 `entity_role`。
- 不为凑数量补标签；宁少打，不弱证据打标。**例外：`ingredient_technology` 反向从宽——字典外成分/技术宁可多打 `other:`，也不漏标（见 4.4.1）。**

### 6.3 证据规则
每个 Agent 判断字段必须有证据，**每个 `other:<名>` 标签也必须有 evidence record，并在 `extracted_name` 写出原文显示名**。优先级：`content` > `title` > `summary` > `raw_tags` > `url` > `source_defaults`。`source_defaults` 只能辅助基础字段，不能单独支撑深度语义标签。

> 后处理约束：`other:<slug>` 是开放词表字段的受控逃逸值，不是正式标签，晋升前不计入正式 taxonomy；缺 `extracted_name` 或证据的 `other:` 标签应被 post-processing 标记复核。封闭枚举字段出现非词表正式标签仍一律拒绝。

### 6.4 与原型"置信度评分"的处理
原型月报要求竞品动态展示"置信度评分"。本字典遵循项目决策 008，**打标阶段不输出 AI 自评 confidence**。月报若需展示置信度，应在**生成阶段**由规则派生（来源性质 + 证据字段数 + 是否官方一手），或以人工复核抽样标注，不污染打标 JSON。
