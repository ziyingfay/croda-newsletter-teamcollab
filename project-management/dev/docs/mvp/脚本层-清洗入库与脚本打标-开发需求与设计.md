# 脚本层（清洗入库 + 脚本打标）· 开发需求与设计构思

> 范围：第一层"清洗/入库脚本"——把抓取的原始记录变成 `rss_clean.json`（基础字段），
> 并补上 V4 字典里下放给脚本的标签（spans + company + ingredient_mentions + is_event/event_type
> + fact_hints）。**不含**打标 Agent（relevance/section），**不含**抓取本体（spider，Alexi 侧）。
>
> 依据：`MVP关键工件契约.md`（rss_clean.json 契约）、`标签字段字典-V4-draft.md`（基础字段 + 脚本字段）、
> `source_profiles_seed.json`、第 1–3 轮测试里已写的原型脚本。

---

## 一、结论先行

- **脚本打标层（spans / company / ingredient / is_event）已有可用原型，且经 3 轮测试验证，成熟度最高，
  可以最先正式开发**——主要工作是把"测试里写死的小词表"改成"配置驱动的完整别名表"。
- **基础字段清洗层（id/url/canonical/published_at/language/region/extraction/source_profiles 派生）
  测试里基本没做**（因为测试输入 `all-articles(1).json` 已是抽取好的正文），但**契约已定义清楚**，
  属于"规范明确、确定性强、需新写"的活，工作量中等。
- **硬依赖**：真实清洗（HTML→正文、去重、raw_ref）依赖 spider 的输出格式，而 spider 目前是空的
  （`product/spider/*/.gitkeep`）。所以**清洗层要等 spider 输出契约定了才能完全落地**；脚本打标层
  **现在就能基于 rss_clean 形态的数据独立开发与测试**。

## 二、现状盘点与成熟度评估

逐字段（基础字段 = MVP 契约 rss_clean.json；脚本字段 = V4 字典下放项）：

| 字段/组件 | 契约/字典要求 | 测试里是否做过 | 成熟度 | 说明 |
|---|---|---|---|---|
| `article_id` | 稳定 hash：canonical_url→url→source+title+date 兜底 | 否（直接用源 id） | 低 | 需实现稳定 hash 规则 |
| `raw_ref` | raw_file + item_index | 否（测试无 raw 流程） | 低 | 有 raw 后 trivial |
| `title`/`summary`/`content` 清洗 | 去空格、解 HTML 实体、去导航/广告 | 否（源已是 contentText） | **低/未做** | 真实抓取需 HTML→正文；测试跳过了 |
| `content_length` | len(content) | 是 | 高 | trivial |
| `url`/`canonical_url` | 去跟踪参、解跳转、canonical | 否 | 低 | 需实现规范化 |
| `published_at` | 多格式解析；缺失置空（不拿抓取时间冒充） | 否（测试已知 publishTime 不可靠） | 低 | 需实现；测试明确标注过此坑 |
| `fetched_at` | 抓取时间透传 | 透传 | 高 | trivial |
| `source_key`/`source`/`source_list`/`ingest_method`/默认 language/region | 命中 `source_profiles` 派生 | 否（测试 matched=false） | **中（有 seed 但不全）** | seed 仅 13 条、覆盖语料 10/27；需补全 + 写匹配逻辑 |
| `content_language` | source 默认 + 检测修正 | 否 | 低 | 需接语言检测 |
| `market_region` | source 默认 + 内容地名规则 | 否 | 低 | 多数靠 source 默认即可 |
| `extraction_status` | success / partial（质量） | 测试用长度近似 | 中 | 契约只要两态；需定义质量判定 |
| `raw_tags` | RSS category 去重 | 否 | 低 | 有 raw 后 trivial |
| **`spans` 切句** | V4 新增，标题+正文按句编号 | **是（正则切句）** | **中-高** | 原型可用；需强化英文/缩写/最小长度 |
| **`company` / `company_normalized`** | NER + Watchlist 别名归一 | **是（28 词 hint 命中）** | **中** | 原型是小词表；production 需全 Watchlist 别名 + 归一化 |
| **`ingredient_mentions`** | 成分别名表命中（仅抽取） | **是（29 词 hint）** | **中** | 需扩成完整字典别名表 |
| **`is_event` / `event_type`** | 活动关键词规则 | **是（规则表）** | **中-高** | 原型可用；词表需可配置维护 |
| **`fact_hints`** | 竞品/客户/成分/活动命中 | **是** | 中 | 测试便利项，可保留或并入上面字段 |

**已有原型位置**：`product/output/test-results/section-tagging-test/scripts/make_pilot_sample.py`
（spans/company/ingredient/is_event/fact_hints 的可运行实现）、`build_probe_candidates.py`（复用同逻辑）。
这些是**测试级原型**（词表写死、无 source_profiles 接入、无真实清洗），不是 production，但**算法骨架可直接抽取复用**。

## 三、设计构思

### 3.1 拆成两个独立模块（解耦、可分别上线）

```text
[spider 输出 raw/rss_raw_*.json]   ← 不在本需求范围（Alexi）
        │
        ▼
模块 A · cleaner（清洗入库）          → product/output/result/spider/<YYYYMM>/<ts>-rss-clean.json
   raw → 基础字段(id/url/canonical/published_at/source_profiles 派生/language/region/extraction)
        │
        ▼
模块 B · script-tagger（脚本打标）    → product/output/待打标.json（= rss_clean + 脚本字段）
   rss_clean → + spans + company(+normalized) + ingredient_mentions + is_event/event_type + fact_hints
        │
        ▼
[打标 Agent 小龙虾1 消费]            ← 不在本需求范围
```

- 两模块通过 `rss_clean.json` 契约解耦：A 产出它，B 消费并增补。
- **B 可以现在就开发**（拿现有 rss_clean 形态数据或 `all-articles` 转一版即可测）；A 的"真实清洗"
  部分等 spider 输出格式定了再补，但 A 的"基础字段规范化/source_profiles 派生"部分不依赖 spider 细节，可先写。

### 3.2 配置驱动（关键设计原则）

把测试里写死的词表全部外置为**配置文件**，由脚本加载，便于运营维护、与字典同源：

| 配置 | 来源 | 用途 |
|---|---|---|
| `source_profiles.json` | 现有 seed（需补全到 ~42 条） | source_key/source/source_list/ingest_method/默认 language/region 派生 |
| `watchlist_entities.json` | V4 字典第六节客户名单 | company 归一化（别名→display_name）+ 角色匹配（后端用，B 只抽取） |
| `ingredient_alias.json` | V4 字典成分/技术词表 + 别名 | ingredient_mentions 命中 |
| `event_keywords.json` | V4 event_type 规则表 | is_event/event_type |

> 原则：**脚本不内嵌业务词表**；改词表 = 改配置，不动代码。这也保证脚本与字典 V4 单一事实源一致。

### 3.3 各字段实现方式（要点）

- `article_id`：`sha1(canonical_url)`；缺 canonical 用规范化 url；再缺用 `source_key|title|published_at`。
  **必须稳定**（同文重处理同 id），是跨工件主键。
- `url/canonical_url`：去 utm/ 跟踪参、解短链跳转、取 canonical link。
- `published_at`：解析 RSS pubDate/published/updated 多格式 → ISO datetime；**解析不到就置空**，
  绝不用 fetched_at 冒充（测试已证 publishTime 不可靠）。
- `source_*` 派生：用 `source_name`/抓取入口匹配 `source_profiles.source_key`，命中即带出
  source/source_list/ingest_method/默认 language/region；**未命中**记 `source_profile_matched=false`
  进 run_log，由运营补 profile（不阻断入库）。
- `content_language`：先取 profile 默认，再用轻量语言检测（如 langid/fasttext）对标题+正文校正。
- `extraction_status`：按清洗后正文质量给 `success`/`partial`（正文足够且少噪声=success；有正文但含导航/广告/偏短=partial）。失败/空正文**不入 articles**，进 fetch_log。
- `spans`：`{id,text}` 按句切（中英文终止符 + 换行），s0=标题；过滤极短碎片；保证 id 稳定连续，供打标 Agent 用 `trigger_span_id` 指针。
- `company`：用 watchlist 别名表做实体匹配（中英文别名→display_name），输出 display 名 + `company_normalized`；**只抽取，不判角色**（角色后端用 watchlist 匹配）。
- `ingredient_mentions`：用 ingredient_alias 表命中（仅抽取词面；主轴显著性留 Agent）。
- `is_event/event_type`：event_keywords 规则首命中定 type；无命中 is_event=false。
- `fact_hints`：竞品/客户/成分/活动命中清单（给短文/空文当线索；可选保留）。

### 3.4 可测试性（每模块配 validator + run_log）

- 复用测试期 validator 模式：写 `validate_rss_clean.py`（校验基础字段齐全/类型/article_id 唯一稳定/
  source 派生一致）与 `validate_article_package.py`（校验脚本字段 + spans 结构 + 与契约一致）。
- 每次运行产出 `run_log.json`（条数、未命中 profile 列表、language 分布、extraction 分布、异常）。
- 用现有 30 篇 + 探针 33 篇做回归 fixture。

## 四、分阶段开发计划

- **P0（可立即做，价值最高）· script-tagger（模块 B）正式化**
  从 `make_pilot_sample.py` 抽出脚本打标逻辑 → 独立脚本 + 4 个外置配置（profiles/watchlist/ingredient/event）
  → 输入 rss_clean，输出 `待打标.json`（rss_clean + 脚本字段）→ 配 validator + run_log。
  依赖：仅需 rss_clean 形态数据（现有可造）。**这是"已够成熟、先开发"的部分。**
- **P1 · cleaner（模块 A）基础字段非抓取部分**
  实现 article_id/url/canonical/published_at/source_profiles 派生/language/region/extraction_status。
  依赖：source_profiles 补全到 ~42 条（运营/PM 配合）；语言检测选型。
- **P2 · cleaner 真实清洗对接 spider**
  HTML→正文抽取、去导航广告、去重(canonical)、raw_ref/raw_tags。依赖：**spider 输出契约定稿**。
- **P3 · 运营化**
  source_profiles/watchlist/词表的维护流程；按月 `YYYYMM` 跑批；与打标 Agent 串联跑通端到端。

## 五、风险与未决

1. **source_profiles 不全**：seed 仅 13 条、覆盖语料 10/27；客户 v260605 已确认 ~42 个信源 → 需先补全
   profile，否则 source/language/region 大量走默认或落空。**P1 的前置。**
2. **spider 输出格式未定**：真实 HTML→正文清洗（P2）卡在这；建议先与 Alexi 对齐 rss_raw 字段契约。
3. **published_at 不可靠**：测试已证；脚本宁可置空也不冒充，月度时间窗过滤的影响需 PM 知悉。
4. **company 抽取精度**：别名表匹配 vs 真 NER——MVP 先别名表（够竞品/KA 这类已知实体）；长尾新公司
   靠 Agent 兜底补（在打标层，不在本脚本）。
5. **中文语言检测/分句边界**：选型（langid/fasttext + 自定义分句）需小测；spans 切句对英文缩写、
   分号、特殊符号要回归测试。
6. **职责边界**：本脚本只"抽取/规范化",不做语义判断（relevance/section/标签显著性都在打标 Agent）。
   不要让脚本越界去"判"，避免重蹈"公式反推"覆辙。
