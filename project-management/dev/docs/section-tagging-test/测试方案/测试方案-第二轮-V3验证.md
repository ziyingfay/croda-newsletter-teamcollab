# 第二轮测试方案 · V3 框架验证

> 承接第一轮 Pilot-30（见 `团队分享-栏目打标测试.md`）。第一轮回答的是"用哪种范式"
> （结论：栏目 AI 一等判断，不用标签公式反推）；第二轮回答的是"V3 这套具体设计**做对了
> 没有、能不能稳定跑、值不值这个成本**"。

## 0. 一句话定位

第一轮是**选型测试**（单评测者、证明公式反推不行）；第二轮是**验证测试**
（多评测者/多模型、证明 V3 的具体改动真的解决了第一轮发现的问题，且可量化、可复跑）。

## 1. 本次测试的具体目的

第一轮只证明了"方向对"，但有三个硬伤没被验证，第二轮专门打这三个：

**目的 A — 验证 V3 的结构性改动是否真的消除了第一轮的失败模式。**
第一轮发现公式反推有四类错（无相关性闸门、活动劫持、竞品名盖故事、汇编推不出）。
V3 用"栏目 AI 直判 + event 拆轴 + 标志位"来修。**必须用数据证明这些错真的没了**，
尤其是 `is_event` 拆轴后，"供应商在 webinar 发新原料"是否不再被误判成 market_event。

**目的 B — 把第一轮循环论证的 1.00 换成真实可信的数字。**
第一轮 B/C/D 满分是"同一个大脑既出标准答案又判断"造成的循环。**破循环靠的是人工
gold standard，不是换模型**：
- **去循环**：标准答案由 **≥2 名人工标注者**独立出（对齐分歧、更新栏目定义），与任何模型无关。
- **被测打标模型**：**Opus 4.8（`claude-opus-4-8`）与 Sonnet** 两个，独立跑、互不可见。
- **编排与打分**：由 Fable 5 负责（不作为被测评测者，避免又当运动员又当裁判）。

由此得到真正的 `primary_section_accuracy`、`per_section_f1`、`confusion_matrix`，以及
**模型间一致性**（同一篇 Opus 与 Sonnet 判同栏目的比例）。Kimi/MiniMax 本轮不接入
（环境无通道）；如需国产模型迁移性，由团队外部跑后把输出导入再合并分析。

**目的 C — 验证 V3 新机制本身可用、可控、值这个钱。**
三件第一轮没法测的新东西：
- **证据指针**：AI 填 `trigger_span_id` 的准确性——指对句子率（`span_valid`）与
  推理成立率（`inference_valid`）。这是 V3 的核心赌注，必须单独评。
- **脚本字段**：脚本抽的 `company`/`ingredient_mentions`/`is_event` 对不对
  （precision/recall vs 人工），决定哪些标签能安全下放。
- **成本与稳定性**：V3（C+D）的真实 token、JSON/schema 失败率、对模糊文章 3× 重跑的
  `section_flip_rate`——验证"取消置信度、靠人工抽检"是否站得住。

> 一句话：第一轮证明"该这么做"，第二轮证明"这么做做对了、跑得稳、花得起"。

## 2. 从大项目测试初衷看，第二轮如何满足初衷

大项目的测试初衷（来自需求与团队讨论）始终是同三问，第二轮逐条对齐：

| 大项目初衷 | 第一轮做到 | 第二轮补齐 |
|---|---|---|
| **能不能把文章准确放进栏目** | 证明公式不行、判断行（但循环） | 多评测者/多模型给出**真实准确率 + 混淆矩阵**，定位还在错的栏目 |
| **AI 打标稳不稳定** | 只看了单次自洽 | **3× 重跑的 flip-rate**（聚焦模糊簇）+ 模型间一致性 |
| **消耗多大、便不便于 Agent 框架** | 估算了相对成本 | **实测 token + schema 失败率**；验证 spans/脚本字段前置是否真的让 Agent 流程更简单 |
| **对后续工作流是否有帮助** | 定性推断 | **下游探针**：搜索召回 + 报告撰写可用性，用数据说话 |
| **标签到底要不要、要哪些** | 提出"脚本事实 vs AI 语义"拆分 | **脚本字段 precision/recall**，用证据决定一/二期标签边界 |

换句话说：第一轮帮我们**选定了方案**，第二轮产出的是**能拿去对客户/对内拍板"一期就这么做"
的量化依据**，以及"哪些标签放二期"的硬证据——正好接上老燕说的"评估 pros/cons、成本收益，
再决定放一期还是二期"。

## 3. 测试范围与数据

- **样本**：先用 Pilot-30（已带 spans 的 `pilot-30-input-v3.json`）跑通 V3 全链路；
  通过后扩到 **120 篇**（沿用第一轮分层：70 完整 / 25 部分 / 25 短空，覆盖行业媒体、PCHi、
  竞品/供应商官号、下游品牌、法规、低值促销）。
- **数据依赖**：等 Alexi 按时间窗口多抓数据后再扩 120；不足则先在 30 上得方法学结论。
- **标注**：≥2 人独立标前 30–50 篇，对齐分歧、更新栏目定义，再标其余（沿用第一轮 R1）。

## 4. 测试矩阵（跑什么）

主对比是 **V3（C+D 混合）vs 第一轮的判断基线**，外加模型维度：

| 维度 | 取值 |
|---|---|
| 框架 | V3（栏目直判 + 脚本字段 + 指针证据）；对照：V2 风格全标签（A，仅在完整正文子集） |
| 模型 | Opus 4.8 与 Sonnet（两个独立打标评测者）；Fable 5 仅编排打分 |
| 重复 | 模糊簇文章 3× 重跑测 flip-rate |

> V3 内部不再纠结 B/C/D 互比——第一轮已说明单评测者下三者栏目判断同源；C/D 的差异
> 只在下游（搜索/报告）和短空文成本上体现，由第 5 节探针单独测。

## 5. 指标（在第一轮基础上新增 V3 专属项）

**栏目质量（对齐目的 A、B）**
- `primary_section_accuracy`、`per_section_precision/recall/f1`、`confusion_matrix`
- `relevance_gate`：错留垃圾 / 错杀有用（一级闸门，单独报）
- **硬案例分项**：event-vs-substance、roundup→market_brief、相关性边界三个小集各自准确率
- **模型间一致性**：同篇不同模型同栏目比例

**证据机制（V3 专属，对齐目的 C）**
- `span_valid`：`trigger_span_id` 指向的句子真实在题（用 validator `--article` 自动校验存在性，
  人工校验在题性）
- `inference_valid`：该句确能支撑判断（人工，**这才是拦"事实对、分析错"的关键指标**）
- `span_id_invalid_rate`：指针指向不存在/越界 span 的比例（应趋 0）

**脚本字段（对齐目的 C、标签边界）**
- `company` / `ingredient_mentions` / `is_event` 的 precision/recall vs 人工

**成本与稳定性**
- 实测 avg input/output tokens（Opus 4.8 与 Sonnet 分别报，回应第一轮 token 估算只是启发式）
- `schema_failure_rate`（用 `validate_tags_v3_draft.py`）
- `section_flip_rate`（模糊簇 3× 重跑）

**下游可用性探针（对齐"对后续工作流是否有帮助"）**
- 搜索探针：`欧莱雅 / PDRN / PCHi` 三查询，用产出标签算召回
- 报告探针：取一个栏目，把文章+标签+指针证据喂给撰写 prompt，判证据是否够写栏目不必回读全文

## 6. 执行步骤

1. 生成 V3 输入包：`make_pilot_sample.py`（已产出 `pilot-30-input-v3.json`，带 spans/脚本字段）。
2. 按 `llm_prompt-v3-draft.md` + `tag_output-v3-draft.schema.json`，每模型独立跑 30 篇。
3. 校验：`validate_tags_v3_draft.py <out> --article pilot-30-input-v3.json`
   （自动查 span 存在性、决策一致性、脚本字段防回流、禁置信度）。
4. 人工对 30 篇评 `inference_valid` 与硬案例分项；脚本字段比对人工标注。
5. 跑通后扩 120 篇，加模糊簇 3× 重跑与下游探针。
6. 结果写入 `analysis/`，并在 `团队分享-栏目打标测试.md` 追加"轮次 2"。

## 7. 通过/淘汰判据

- **目的 A 通过**：event-vs-substance 小集准确率显著高于第一轮公式（基本不再误判 market_event）；
  相关性闸门错留垃圾接近 0。
- **目的 B 通过**：跨模型 `primary_section_accuracy` 达成可接受阈值（先以人工标注一致性为天花板参照），
  混淆集中在已知模糊簇而非系统性偏向。
- **目的 C 通过**：`inference_valid` 高、`span_id_invalid_rate`≈0、脚本字段 precision/recall 达标、
  schema 失败率低、flip-rate 仅集中在模糊簇。
- **任一不达标**：回到对应改动（如 event 拆轴没解决劫持 → 重审优先级规则；指针错多 →
  回退到"指针+短引文"混合），不扩 120，先修。

## 8. 前置依赖与风险

- **数据量**（陈炜提示）：30 篇分布不均，价值链/部分栏目可能样本太少，效果"好不一定真好、
  差比较确定"。故 30 只做方法学验证，结论以 120 为准，并请 Alexi 按时间窗口多抓。
- **模型可用性**：本环境只能跑 Claude 家族；Opus 4.8 与 Sonnet 可经 Agent/Workflow
  派发，Kimi/MiniMax 无通道，需团队外部跑后导入。
- **人工标注成本**：`inference_valid` 与硬案例需人读，控制在 30 篇先做，再决定是否全量。
  注意：人工 gold（破循环用）与生产期人工抽检是两件事——前者是测试的答案卷，后者是
  上线后的质量抽查，不要混为一谈。

## 9. 规模化与生产期监控（前瞻，非第二轮执行项）

回应"几千篇时还稳吗"。**单篇判断准确率不随数据量下降**（每篇是独立调用），但**聚合
准确率会随长尾覆盖变广而下漂**——30/120 没见过的文章类型到几千篇一定出现。规模化的承诺
不是"每篇都对"，而是"**错误率有上界，且不确定/新颖的被复核队列接住**"。因此：

- **不靠无限加测**：人工只标几百篇（每栏目≥20–30 例）出指标即可。
- **几千篇的用法**：跑模型但**不全人工标**，只做①栏目分布审计（是否畸形地全压一个栏目）、
  ②抽查 `needs_review` / 低证据 / `other:` 新词那批。
- **生产期监控指标**：needs_review 率、人工抽检命中错误率、月度栏目分布漂移、`other:` 增长曲线。

> 本节属于"上线后怎么知道还稳"，与第二轮"上线前测得多准"是两条轴；列在此处备忘，
> 不增加第二轮带标注实验的工作量。
