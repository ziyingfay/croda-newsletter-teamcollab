# SQLite 数据库数据流说明

本文档用于说明 RSS 抓取数据进入 SQLite 数据库后，在各张表之间如何流转，以及 AI Agent 打标签时会读取和写入哪些数据。

---

## 一、整体数据流

一句话概括：

```text
RSS JSON
→ 记录导入批次
→ 存文章基础信息
→ 存正文和抽取质量
→ 标记为待打标
→ OpenClaw 读取待打标文章
→ 写入打标批次
→ 写入正式标签和证据
→ 写入低置信候选或 Other 候选
→ 根据公司 watchlist 支持公司标签和历史回填
```

对应表：

```text
rss_import_runs
→ articles
→ article_contents
→ article_tag_status
→ v_articles_for_tagging
→ tagging_runs
→ article_tags
→ tag_evidence
→ candidate_tags / other_candidates
→ company_watchlist
```

---

## 二、RSS 抓取后：进入导入批次表

RSS 抓取脚本先生成原始 JSON 文件。每次把一个 JSON 文件导入数据库时，先在 `rss_import_runs` 里记录一次导入任务。

### 表：`rss_import_runs`

这张表记录“这一次导入”的运行信息，例如：

- 导入的是哪个 JSON 文件
- 文件 hash 是什么
- 一共有多少条文章
- 成功插入多少条
- 更新多少条
- 重复多少条
- 失败多少条
- 导入开始和结束时间
- 导入状态

它不负责存文章内容，只负责记录导入批次。

---

## 三、每篇文章：进入文章基础表

每条 RSS item 清洗后进入 `articles`。

### 表：`articles`

这张表存文章的基础事实，例如：

- `article_id`
- `url`
- `canonical_url`
- `title`
- `summary`
- `published_at`
- `fetched_at`
- `source_key`
- `source_name_snapshot`
- `raw_json`

这一层的核心作用是建立“文章身份”。

去重也主要发生在这里。例如同一篇文章的 URL 已经存在，就不重复创建新文章。

注意：`articles` 不直接存多值标签。它只保存文章本身的信息。

---

## 四、正文抽取后：进入正文表

如果后续有正文抽取结果，就写入 `article_contents`。

### 表：`article_contents`

这张表存正文和正文质量相关信息，例如：

- `content`
- `content_length`
- `content_hash`
- `extraction_status`
- `quality_score`
- `raw_tags_json`
- `wechat_original_url`
- `extracted_at`
- `extractor_version`

正文和文章基础信息分开存，是因为正文通常很长。Agent 批量查看待处理文章时，不一定每次都要加载全文。

---

## 五、新文章准备给 AI：进入打标状态表

每篇新文章入库后，会在 `article_tag_status` 里创建一条状态记录。

### 表：`article_tag_status`

这张表相当于 OpenClaw 的任务队列表。

示例：

```text
article_id = xxx
status = pending
priority = 0
```

常见状态包括：

- `pending`：等待打标
- `in_progress`：正在打标
- `tagged`：已完成打标
- `needs_review`：需要人工复核
- `failed`：打标失败

OpenClaw 后续会根据这张表找到需要处理的文章。

---

## 六、OpenClaw 读取文章：通过统一视图

OpenClaw 不需要自己拼很多张表。建议让它读取一个统一视图。

### 视图：`v_articles_for_tagging`

这个视图会把以下信息拼在一起：

- `articles`：标题、URL、摘要、发布时间、来源
- `article_contents`：正文、正文长度、抽取状态、质量分
- `source_profiles`：来源信息、抓取方式、来源性质默认提示
- `article_tag_status`：当前打标状态、优先级、最近一次打标 run

这样 Agent 一次查询就能拿到完整上下文。

典型查询逻辑：

```sql
SELECT *
FROM v_articles_for_tagging
WHERE tag_status = 'pending'
ORDER BY priority DESC, published_at DESC
LIMIT 20;
```

---

## 七、Agent 开始打标：创建打标批次

OpenClaw 每次开始一批打标任务时，先写入 `tagging_runs`。

### 表：`tagging_runs`

这张表记录一次 AI 打标任务，例如：

- 哪个 Agent 执行
- 使用哪个模型
- 使用哪个标签字典版本
- 使用哪个公司 watchlist 版本
- 本次是初次打标、重打标，还是公司回填
- 本批处理多少篇文章
- 成功多少篇
- 失败多少篇
- 开始和结束时间

它相当于 AI 打标任务日志。

---

## 八、Agent 输出正式标签：进入正式标签表

通过校验后的正式标签写入 `article_tags`。

### 表：`article_tags`

这是打完标签后的主存储位置。

例如一篇文章被打成：

```text
source_nature = media_report_interview
topic = sustainability
topic = supply_chain
product_category = functional_ingredients
company_tags = nestle
```

这些标签不会塞回 `articles`，而是一条标签一行地存在 `article_tags`。

这样以后要查：

```text
所有 topic = sustainability 的文章
```

就可以直接查 `article_tags`，再关联回 `articles`。

---

## 九、每个标签的理由：进入证据表

每个正式标签都应该有证据，写入 `tag_evidence`。

### 表：`tag_evidence`

示例：

```text
field = topic
label = sustainability
evidence_field = summary
evidence_text = "regenerative agriculture scales amid climate risks"
reason = "文章核心讨论再生农业和气候风险"
confidence = 0.88
```

这张表非常关键。

它让我们之后可以回答：

```text
AI 为什么给这篇文章打了这个标签？
```

而不是只看到一个无法解释的标签结果。

---

## 十、低置信但可能有用：进入候选标签表

如果 Agent 觉得某个标签可能合适，但置信度不够，就不要进入正式标签。

### 表：`candidate_tags`

比如某个标签置信度只有 0.65，可以写入 `candidate_tags`。

它表示：

```text
这个标签可能对，但还需要人工或后续规则确认。
```

这可以避免把不确定判断直接污染正式标签结果。

---

## 十一、词表没有合适标签：进入 Other 候选表

如果文章明确有一个新概念，但当前标签字典没有合适标签，就进入 `other_candidates`。

### 表：`other_candidates`

例如文章多次讨论：

```text
cultivated cocoa butter
```

但当前词表里只有更宽泛的：

```text
cultivated_food
```

那么 Agent 可以把 `cultivated_cocoa_butter` 写入 `other_candidates`，等待人工审核。

这张表服务“活字典”机制。

Other 候选不会马上变成正式标签，而是等待每周 qualification：

- `approved`：成为正式词表
- `merged`：合并到已有标签
- `rejected`：拒绝

---

## 十二、公司标签：受公司监测表控制

公司名不能由 Agent 自由生成，而是先看 `company_watchlist`。

### 表：`company_watchlist`

例如当前启用：

- `nestle`
- `coca_cola`
- `pepsico`
- `unilever`
- `danone`
- `mondelez`
- `cargill`

Agent 只有在文章明确涉及这些公司，并且公司是文章主体或重要相关方时，才写入正式标签表：

```text
field = company_tags
label = coca_cola
```

如果以后客户新增：

```text
givaudan
```

就先把它写入 `company_watchlist`，再触发历史文章回填。

---

## 十三、为什么不要把标签直接放进 articles？

不建议把标签直接存成：

```text
articles.tags = ["sustainability", "supply_chain"]
```

原因是：

1. 一篇文章会有多个字段的多个标签。
2. 每个标签需要置信度。
3. 每个标签需要证据。
4. 后续会有重打标。
5. 标签字典会更新。
6. 公司 watchlist 会变化。
7. Other 候选需要审核。

如果全部塞进 `articles`，后续查询、复核、回填都会很麻烦。

更好的方式是：

```text
articles：只存文章事实
article_tags：存正式标签
tag_evidence：存标签证据
candidate_tags：存低置信候选
other_candidates：存新标签候选
company_watchlist：控制公司实体标签
```

---

## 十四、最终设计思想

这个数据库的核心不是“存 RSS 文章”，而是服务一套可追溯的 AI 打标流程。

最重要的分层是：

- 文章事实和 AI 判断分开。
- 正式标签和候选标签分开。
- 标签结论和标签证据分开。
- 普通标签和公司监测名单分开。
- 当前结果和历史打标 run 分开。

这样后续做筛选、复核、重打标、扩展词表、客户公司监测，都会比较稳。
